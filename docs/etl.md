## Overview

The ETL pipeline syncs on-chain Stellar transactions into the `transactions` table.
It runs on a configurable schedule via `ETLWorker` and delegates the actual work to
`TransactionETLService`.

---

## Idempotency guarantees

### 1. Batch-level deduplication (`etl_batches` table)

Every scheduled tick generates a **UUID batch ID** before any work starts.
`TransactionETLService.runETL(signal, batchId)` checks the `etl_batches` table:

| Scenario | Behaviour |
|---|---|
| `batch_id` not found | Create row (`pending`), proceed normally |
| `batch_id` found with `status = completed` | Return cached result immediately â€“ **no re-processing** |
| `batch_id` found with `status = failed/running` | Row already exists; skip `create`, call `markRunning`, retry the work |

This means a crashed or retried run with the **same batch ID** will never
double-count analytics.

### 2. Row-level deduplication (`tx_hash` UNIQUE constraint)

`saveTransactions` uses `INSERT â€¦ ON CONFLICT (tx_hash) DO NOTHING`.
Even if the batch-level guard is bypassed (e.g. a manual backfill), individual
transactions are never inserted twice.

### 3. Input validation

`TransactionETLService.isValidTxHash(hash)` rejects any `tx_hash` that does not
match `/^[0-9a-fA-F]{1,128}$/`.  Rows with invalid hashes are skipped and counted
as `transactionsSkipped`, preventing SQL injection via ETL inputs.

## Batch lifecycle


State transitions are conditional (`WHERE status NOT IN ('completed', 'failed')`)

## `etl_batches` table schema

| Column | Type | Description |
|---|---|---|
| `batch_id` | UUID PK | Stable identifier generated by ETLWorker |
| `status` | enum | `pending / running / completed / failed` |
| `operations_fetched` | integer | Total Horizon operations fetched |
| `transactions_inserted` | integer | New rows written to `transactions` |

| `error_message` | text | Populated on failure |

## ETLWorker

`ETLWorker` manages the schedule and graceful shutdown:

- Calls `start(intervalMinutes?)` to begin periodic runs (default: 5 min).
- Generates a **fresh UUID** for each tick and passes it to `runETL`.
- On `stop()`: clears the interval, sends an `AbortSignal` to the in-flight run,
  and waits up to `drainTimeoutMs` (default 30 s) for it to finish.

```ts
import { ETLWorker } from './services/etlWorker.js'

const worker = new ETLWorker(config)
worker.start(5)          // run every 5 minutes
await worker.stop()      // graceful shutdown
```

---

## Retry behaviour

`ETLWorker` does **not** automatically retry a failed batch with the same ID.
Each new tick generates a new UUID, so a failed batch is simply superseded by
the next scheduled run.

For manual retries (e.g. ops tooling), pass the **same** `batchId` to
`runETL` â€“ the service will skip the work if it already completed, or resume
from scratch if it failed.

---

## Security

- `tx_hash` values are validated against a hex-only regex before any DB write.
- No user-supplied strings are interpolated into raw SQL; all queries use
  Knex parameterised bindings.
- Export paths are not derived from ETL inputs, so path traversal is not
  possible.

## Vault State Reconciliation

### Overview

The vault reconciliation job compares persisted vault state in the database with on-chain state from the Soroban contract to detect drift caused by missed Horizon events or failed ETL batches. This is a **read-only** operation that reports anomalies without auto-correcting them.

### Reconciliation Job

**Job Type**: `vault.reconcile`

**Handler**: `TransactionETLService.reconcileVaults()`

**Schedule**: Run periodically (e.g., daily or hourly) via the job system.

### How It Works

1. **Fetches persisted vaults** from the `vaults` table (all vaults or a subset via `vaultIds` option)
2. **Reads on-chain state** via `soroban.getVault()` for each vault in batches
3. **Compares key fields**:
   - `status` (normalized for case-insensitive comparison)
   - `amount`
   - `verifier` address
   - `success_destination` address
   - `failure_destination` address
4. **Reports drift** via structured logs using the abuse-monitor taxonomy:
   - `vault.vault_missing_onchain` - vault exists in DB but not on-chain
   - `vault.vault_state_drift` - one or more fields differ between DB and on-chain
   - `vault.vault_reconciliation_error` - error during reconciliation (e.g., RPC timeout)

### Job Payload

```typescript
interface VaultReconcileJobPayload {
  vaultIds?: string[]    // Optional: specific vault IDs to reconcile
  batchSize?: number     // Optional: batch size for RPC calls (default: 50)
}
```

### Example Usage

```ts
// Enqueue a reconciliation job for all vaults
await enqueueJob('vault.reconcile', { batchSize: 50 })

// Enqueue a reconciliation job for specific vaults
await enqueueJob('vault.reconcile', { vaultIds: ['vault-1', 'vault-2'] })
```

### Result

The reconciliation returns:

```typescript
{
  totalVaults: number      // Total vaults to reconcile
  checked: number          // Successfully checked vaults
  driftDetected: number    // Vaults with state drift
  missingOnChain: number   // Vaults missing on-chain
  errors: number           // Reconciliation errors
}
```

### Idempotency and Resumability

- **Idempotent**: Running the same reconciliation multiple times produces consistent results
- **Resumable**: If aborted via `AbortSignal`, the job can be re-run; no partial state is persisted
- **Bounded**: Processes vaults in configurable batches to avoid RPC rate limiting

### Configuration

Requires Soroban configuration (same as vault submission):

- `SOROBAN_CONTRACT_ID`
- `SOROBAN_NETWORK_PASSPHRASE`
- `SOROBAN_SOURCE_ACCOUNT`
- `SOROBAN_RPC_URL`
- `SOROBAN_SECRET_KEY`

If Soroban is not configured, the job skips reconciliation and returns zero counts.

### Drift Response

When drift is detected:

1. **Log the anomaly** with full context (vault ID, drifted fields, persisted vs on-chain values)
2. **Do NOT auto-correct** - manual intervention is required to replay missed events
3. **Alert operators** via monitoring systems that consume the structured logs
